## Section 1 — Executive summary

**The three evaluations converge on one uncomfortable conclusion: this bot cannot be trusted to know whether its own orders executed, whether its positions match the broker, or whether its risk controls are active.** Claude Code framed the problem strategically (strategies aren't replayable, no monitoring, no CI), GPT Codex v1 framed it as wiring fraud (shadow modules never called, HALT file is cosmetic theater, 21 `except: pass` blocks, daily reset bug, orphan YAML), and GPT Codex v2 framed it as execution-layer capital extinction (silent OIF write failure, no position persistence, no fill ACK, WebSocket dies silently, AI filter fails open). All three agree on the cross-cutting pathologies: **silent exception swallowing, non-atomic OIF writes, in-memory-only state, missing heartbeats, hardcoded magic numbers, and dashboard/config races**.

Where they disagree is sequencing. Claude Code says "validate strategies first" (4 hours to extract pure-function strategies, then decay monitor). Codex v1 and v2 both say "fix plumbing first" because no amount of strategy work matters if the executor can phantom-position you, double-fire orders, or silently drop trades. **I arbitrate in favor of plumbing-first, unambiguously.** Evidence: the bot's realized -$1,227 over 697 trades is partly strategy edge (-R:R compression from `ema_dom_exit`) but a non-trivial fraction is almost certainly execution leakage — silent OIF write failures produce phantom positions that lock state forever, and TOCTOU double-fires on coincident 1m+5m bar closes are asymmetric-loss events. You cannot even *measure* strategy edge cleanly until the transport is deterministic.

**Update — P0 completed 2026-04-18.** The NT8 data folder has been successfully migrated out of OneDrive to local `C:\Users\Trading PC\Documents\NinjaTrader 8\`. This is the single highest-ROI intervention in the entire plan. The Codex v2 report estimated current risk at **-$500 to -$1,000/month**; a non-trivial fraction of that was OneDrive amplification of code bugs. Atomic writes, file locks, and fill ACK polling now operate on a local NTFS filesystem where they can actually be trusted. The migration was executed cleanly over 7 commits on `feature/knowledge-injection-systems` with all 61 existing tests still passing and a new startup path-validation guard added as a ride-along hardening.

**A finding surfaced during the migration itself that updates P21:** the bot's test suite has been running 61 tests under `python -m unittest discover`, but pytest-style fixture-using tests (6 additional tests, including critical position-manager P&L assertions) are invisible to that runner. Actual test confidence has been ~10% lower than reported. P21 expands accordingly to mandate pytest as the canonical runner.

### Tier 1 — CRITICAL (capital-extinction class, week 1)

**P1. Atomic OIF write with retry + fallback path.** Severity: CRITICAL. Sources: Codex v2. Est: 2h. Capital impact: silent OIF failure = phantom position = runaway loss; single-incident dollar impact on MNQ can be 5–50 ticks ($10–$100) plus psychological tilt; frequency once OneDrive is fixed drops to maybe 1–2/month, but catastrophic when it happens → ~$50–$150/month risk. ROI: very high. **Implementation:** Write to sibling `.tmp` file on the same NTFS volume as `incoming\`, `flush() + os.fsync(fd)`, then `os.replace(tmp, final)` (atomic on same volume via Win32 `MoveFileExW` with `MOVEFILE_REPLACE_EXISTING`). Wrap in a retry loop (5 attempts, 50ms → 800ms exponential backoff) catching `PermissionError` from transient handle holds. Do **not** use `atomicwrites` (deprecated by its maintainer; pytest removed it in issue #10114) or `shutil.move` (copy+delete cross-volume, not atomic). Use stdlib only: `tempfile.mkstemp(dir=staging_same_volume)` + `os.fdopen` + `os.fsync` + `os.replace`. Use a unique monotonic filename per order (`oif_<epoch_ms>_<seq>.txt`) — NT8 docs explicitly warn about lock races on filename reuse. Add a `FILE_FALLBACK_PATH` under `%LOCALAPPDATA%\JenBot\failed_oifs\` where any write that fails all retries is dumped for human inspection, with a Telegram HALT alert.

**P2. Fill acknowledgement gate before advancing Python state.** Severity: CRITICAL. Sources: Codex v2. Est: 3h. Capital impact: Python thinks filled, NT8 is flat (or vice versa) = re-entry on top of existing position; single incident can 2×–4× intended exposure at $2/tick × 25 ticks = $50–$200 loss; 1–3/month → ~$100–$400/month risk. ROI: very high. **Implementation:** Research confirms NT8 ATI **does produce outgoing fill files** — this was not in any evaluation. Documented at `ninjatrader.com/support/helpguides/nt8/information_update_files.htm`: every PLACE OIF that includes a unique `ORDER ID` (11th semicolon field) generates `outgoing\<orderId>.txt` with content `State;FilledQty;AvgFillPrice` (states: Initialized/Submitted/Accepted/Working/Filled/PartFilled/Cancelled/Rejected). Also generates `<Instrument>_<Account>_Position.txt` with `LONG|SHORT|FLAT;qty;avg_px`. Pattern: after atomic OIF write, poll `outgoing\<orderId>.txt` at 20ms cadence with 30-second deadline; on `Filled` update SQLite, advance state, open Python position. On `Cancelled|Rejected` raise and mark intent failed. On timeout: emit `FILL_TIMEOUT` Telegram alert, flag intent as `UNKNOWN_REQUIRES_HUMAN`, set HALT. **Caveat** (forum thread 3540): sub-second lag between fill-state file and position file — trust the per-order file, not the aggregate position file. The `OIF_OUTGOING` constant is already defined in `config/settings.py` as of the P0 migration commits. Use `watchdog` (ReadDirectoryChangesW-backed) or a simple 20ms poll loop; polling is actually simpler and more robust against FSW event coalescing.

**P3. Wire HALT enforcement + circuit breakers at strategy entry.** Severity: CRITICAL. Sources: Codex v1. Est: 0.5h. Capital impact: the HALT marker file is currently theater — creating `.HALT` does nothing; on a bad-data-feed or flash-crash day this is the line between a -$50 day and a -$300 day → ~$50–$150/month risk. ROI: extremely high (shortest implementation). **Implementation:** Add at the top of `_evaluate_strategies()`: `if circuit_breakers.should_halt(): return` and also check `if os.path.exists(HALT_MARKER): return`. Call `circuit_breakers.record_slippage()` and `record_trade_outcome()` in the `_on_trade_closed` handler (currently none of these are called). The HALT marker enforcement is a single `if` at the entrypoint — this is the single highest-ROI patch in the entire plan.

**P4. Position state persistence via SQLite WAL.** Severity: CRITICAL. Sources: Codex v2 (primary), Claude Code (implied by replay needs). Est: 4h. Capital impact: crash-restart produces invisible divergence from NT8 broker state; recovery typically requires manual intervention and has caused entire-account wipeouts in retail bot communities → ~$50–$200/month risk. ROI: very high. **Implementation:** stdlib `sqlite3`, open once on startup with `PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL; PRAGMA busy_timeout=5000;` — note WAL mode is persistent after first set. Schema: `intents(client_order_id PK, strategy_id, symbol, side, qty, bar_close_ts, status, oif_write_ok, created_at, updated_at)` + `positions(symbol PK, qty, avg_price, last_reconciled_at)`. Wrap every state transition in `BEGIN IMMEDIATE`. Use one dedicated `state_writer` asyncio task consuming a `Queue` of events — eliminates concurrency ambiguity for the 5-position scale. Alternatives considered and rejected: `diskcache` (loses relational queries needed for reconciliation), `sqlitedict` (pickle fragility), TinyDB (slow full-file rewrites), plain JSON (concurrency hell). On startup run `reconcile()`: read NT8 position file, read SQLite intents with status in {SUBMITTED, WORKING}, diff, update. Follow the ib_insync `connectAsync` pattern: re-query broker truth on every reconnect; treat DB as truth for intents-not-yet-confirmed only.

**P4b. Exit-path collision: single-decision exit function with priority ordering.** Severity: CRITICAL. Sources: Codex v2 holes #15, #16. Est: 1h. Capital impact: scale-out + stall detector both firing same tick leaves $20–$30 on the table per incident; gap bars crossing both stop and target produce ambiguous NT8 fills worth $10–$15 slippage; combined ~$30–$80/month on MNQ during choppy days. ROI: high. **Implementation:** Replace the current parallel-check pattern (where `check_scale_out()`, `check_stall()`, `check_stop()`, `check_target()` all fire independently on the same bar close) with a single `decide_exit(market, pos) -> Optional[ExitDecision]` function that returns exactly one decision with priority ordering: **stop > target > stall > scale-out**. On gap bars where price crosses both stop and target, prefer stop (conservative risk accounting — assume the adverse fill). The scale-out case is special: if `decide_exit()` returns `scale_out`, the remaining position's exit logic should use a "post-scale-out" parameter set (tighter stop, no stall-detector participation until a new bar arrives), preventing the "stall detector kills orphaned 1x" failure Codex v2 called out. Add unit tests for gap-bar edge cases and scale-out → stall handoff.

**P5. Order idempotency — unique ClOrdID + dedup set + position-precondition.** Severity: CRITICAL. Sources: Codex v1 (OIF dedup), Codex v2 (implied). Est: 2h. Capital impact: double-fire on coincident 1m+5m bar close or `_evaluate_strategies` re-entry = instant 2× intended size; frequency ~1/month but expensive → ~$50–$150/month risk. ROI: very high. **Implementation:** Three independent layers per FIX 4.4 + Alpaca patterns. (a) Generate `client_order_id = f"{strategy}-{YYYYMMDD}-{bar_close_epoch}-{uuid4().hex[:6]}"` and use as the OIF filename so NT8 never sees duplicates. (b) In-memory `OrderedDict[hash, ts]` with 60s TTL capped at 1024 entries, where `hash = sha1(f"{strategy}|{symbol}|{side}|{qty}|{bar_close_ts}")`; drop on hit. (c) Position-state precondition: before any OIF write, compute `delta = intended_target - (broker_pos + working_qty)` and no-op on zero. This mirrors QuantConnect LEAN's `OrderTicket` discipline and Alpaca's "duplicate client_order_id raises" behavior. Persist the high-water counter to SQLite so restart doesn't collide.

**P6. WebSocket heartbeat + staleness detector + NT8 indicator heartbeat file + tick buffer timestamps.** Severity: CRITICAL. Sources: Codex v1 (MQBridge heartbeat), Codex v2 (keep-alive timeout + hole #13 tick buffer age). Est: 2.5h. Capital impact: silent disconnects on loopback don't trigger TCP keepalive (default Windows `KeepAliveTime` is 2 hours); stale bid/ask fed to strategy on dead feed = wrong-side entries; reconnects replaying 90s-old buffered ticks produce "bot trades 90s behind real market" → ~$30–$100/month risk. ROI: high. **Implementation:** Four independent layers, each catches what the others miss. (1) Use `websockets` library with `ping_interval=10, ping_timeout=10`. Wrap every `ws.recv()` in `asyncio.wait_for(ws.recv(), timeout=30)`. Maintain `state.last_data_ts = time.monotonic()`; run an independent `watchdog()` task checking `monotonic() - last_data_ts > 30` every 5s. On breach: Telegram alert `STALE_FEED`, close ws, exponential-backoff reconnect `min(60, 2**n) + random()`. (2) For raw TCP to NT8, use `sock.ioctl(socket.SIO_KEEPALIVE_VALS, (1, 10000, 3000))` on Windows. (3) Most importantly: on loopback 127.0.0.1 TCP keepalive is nearly useless because there's no real network — the C# indicator must write a heartbeat file every 1s; Python checks mtime age. This is the only layer that detects "WebSocket up but NT8 indicator thread dead." (4) **Added from Codex v2 hole #13:** every tick in the bridge's ring buffer gets a `bridge_received_ts` field stamped at receive time. On bot reconnect, the bridge sends the buffer with timestamps intact; the bot rejects any replayed tick where `now - bridge_received_ts > 10s` rather than blindly processing a 90-second-old buffer. This closes the "reconnect latency amplifier" failure mode where strategies fire signals on stale ticks and immediately stop-out on fresh price.

**P6b. Asyncio queue overflow instrumentation.** Severity: CRITICAL. Sources: Codex v2 hole #7. Est: 1h. Capital impact: during OPEN_MOMENTUM bursts (100+ ticks/sec for MNQ on NQ-parent news), the 1024-slot `asyncio.Queue` fills in ~10 seconds and new ticks silently drop with no warning. Bar closes fire on a latent dataset; 9:00 AM signal fires off old data, market has already reversed → $40+ per saturated setup, ~1–2/month → ~$40–$80/month. ROI: high. **Implementation:** Don't rely on `asyncio.Queue`'s default blocking `put()` (which backs pressure but hides it). Instead: (a) instrument queue depth — sample every 100ms, log WARNING at 50% full (512 items), log ERROR + Telegram at 90% full (921 items), record depth timeseries in a small rolling `deque`. (b) Use `put_nowait()` with explicit `asyncio.QueueFull` handling: on full, emit a CRITICAL log with tick payload, drop the tick (don't block the producer which backs up the whole WebSocket reader). (c) Raise `maxsize` to 8192 for the production queue — MNQ burst rate is ~200 ticks/sec max observed, so 8192 is ~40 seconds of headroom before drop. (d) Add a consumer lag metric: `lag_ms = time.monotonic() - tick.bridge_received_ts` at consume time; p95 over 1-minute window; alert if p95 > 500ms. The existing "bot seems slow today" feeling becomes a quantified dashboard number. This patch is how you notice your bot is drowning in ticks before it makes a losing trade on stale data.

**P9. Silent-exception elimination + error-rate monitor.** Severity: HIGH. Sources: Claude Code, Codex v1 (21 `except: pass`), Codex v2 (strategy error swallowing). Est: 3h. Capital impact: silent strategy death means bot stops trading (opportunity loss ~$20–$50/month) but more dangerously, silently corrupted state (~$50–$100/month). ROI: high. **Implementation:** Install `loguru`. Replace every `except: pass` with either `except SpecificError as e: logger.exception(...)` or wrap whole functions with `@logger.catch(reraise=False, message="...")`. Add a Loguru sink with `rotation="50 MB", retention="14 days", diagnose=False, backtrace=True, serialize=True` (JSON lines). Add a `deque`-based rolling-window monitor at ERROR level: >5 errors in 60s → Telegram with 5-minute cooldown. Add `sentry_sdk.init(dsn=..., traces_sample_rate=0.0, environment="prod")` on the free tier (5K errors/month, sufficient; set DSN daily rate-limit ~170 events/day to avoid burning quota on a bad deploy). Add healthchecks.io dead-man switch: async task pings `https://hc-ping.com/<uuid>` every 60s, configured with Period=5min/Grace=2min. These three layers are complementary: in-process catches rates, Sentry catches exception details, healthchecks catches total-crash.

**P10. Wire shadow modules via `_on_trade_closed` handler.** Severity: HIGH. Sources: Codex v1 (explicit), Claude Code (implied). Est: 1.5h. Capital impact: currently you have zero visibility into decay — bot was -$1,227 before you noticed; this is the infrastructure to catch the next drift at -$150 instead of -$1,227 → ~$100–$300/month of prevented drawdown deepening once strategies are producing usable signals. ROI: high. **Implementation:** Add a single `_on_trade_closed(trade)` method to BaseBot that calls `decay_monitor.record_trade(trade)`, `tca_tracker.record_fill(trade)`, `circuit_breakers.record_slippage(trade.slip)`, `circuit_breakers.record_trade_outcome(trade.pnl)`, `sweep_watcher.track_pivot_break(trade.context)`. These methods already exist — they're just never called. This is literally a six-line patch that converts the entire shadow-monitoring suite from fraud to working.

**P11. AI filter safe-default REJECT on timeout.** Severity: HIGH. Sources: Codex v2. Est: 0.25h. Capital impact: Gemini stall on a volatile day = bot trades through a filter that was supposed to stop it. A single filter-bypass incident can be $20–$100 → ~$30/month. ROI: extremely high (15-minute fix). **Implementation:** Change one line. `try: return asyncio.wait_for(gemini_call(), timeout=5) except (asyncio.TimeoutError, Exception): return AIFilterResult.REJECT` (default to blocking, not clearing). Log rejects separately so you can audit false-rejects later.

**P12. File locking on shared `tca_history.jsonl`.** Severity: HIGH. Sources: Codex v1. Est: 1h. Capital impact: corrupted TCA history = wrong slippage metrics = wrong sizing decisions = indirect; also eventually corrupt JSONL parser dies = silent monitoring loss. ~$20–$50/month. ROI: moderate. **Implementation:** Use `filelock` (tox-dev/filelock, used by pipx/poetry/black) — the standard pick over portalocker for this use case. **Critical:** lock a sibling `.lock` sentinel file, not the data file itself (explicit filelock README guidance). Move the file OUT of OneDrive to `%LOCALAPPDATA%\JenBot\tca_history.jsonl` — **the P0 migration already handles this for NT8-related paths, but if tca_history.jsonl is in a project folder that's still in OneDrive, relocate it now.** O_APPEND through Python's MSVCRT is **not** atomic on Windows above ~1KB writes (empirical: notthewizard.com/2014/06/17; CRT doesn't use `FILE_APPEND_DATA` native semantics). Also optionally acquire a Win32 named mutex at startup (`win32event.CreateMutex(None, True, r"Global\JenMNQBot_Executor")`) to enforce single-instance.

**P13. OIF idempotency guard with `threading.Lock` (complements P5).** Severity: HIGH. Sources: Codex v1. Est: 0.5h. Capital impact: bundled with P5. ROI: high. **Implementation:** Add a module-level `_oif_lock = threading.Lock()` around the OIF-write critical section; inside the lock perform the dedup check + position-state check + `os.replace`. Prevents TOCTOU between `is_flat` check and OIF emit.

**P14. Telegram HTML escape all dynamic strings.** Severity: HIGH. Sources: Codex v1. Est: 0.5h. Capital impact: exit reason containing `<` silently drops the alert = you miss the notification = delayed human response. Low dollar but high visibility-risk. ~$10/month. ROI: moderate. **Implementation:** Wrap all `notify_entry`/`notify_exit`/`notify_alert` with `html.escape(text, quote=True)` on every dynamic field. Or switch `parse_mode=HTML` to `parse_mode=MarkdownV2` but that has even more escape edge-cases; HTML + `html.escape` is simpler.

### Tier 3 — MEDIUM (quality/risk reduction, week 2)

**P15. Extract strategies into pure functions for replayability.** Severity: MEDIUM (deferred from Claude's top-3 because plumbing comes first). Sources: Claude Code. Est: 4h. Capital impact: unlocks backtesting → quantifiable strategy edge assessment; indirectly saves whatever strategy-edge-erosion you'd otherwise accept blind → ~$80–$200/month over 3 months. ROI: high long-term. **Implementation:** Define `@dataclass(frozen=True) class StrategySignal(action, size, stop, target, reason)`. Target pure signature `evaluate(bars: pd.DataFrame, snapshot: dict, session: SessionContext, params: StrategyParams) -> StrategySignal`. Strategy never touches `self.bot`. Introduce four `typing.Protocol` ports: `DataReader`, `Clock`, `RiskBudget`, `Executor`. Migrate incrementally: keep old strategy class as 3-line adapter calling the new function. Start with `VWAPPullback` (simplest). Borrow patterns (not code) from nautilus_trader (`Strategy`/`Actor` separation, `LiveClock` vs `TestClock`) and backtrader (Cerebro injection). Add a JSONL tick recorder to BaseBot *immediately* even before the harness exists — you need the corpus accumulating. Rule of thumb: if `evaluate()` needs >6 args, you have hidden state that belongs in `session`.

**P16. Magic-number → Pydantic-validated YAML migration.** Severity: MEDIUM. Sources: Codex v1 (unvalidated weights, hardcoded multipliers), research topic M. Est: 4h. Capital impact: fat-finger YAML values currently silently accepted; validated config prevents ~$20–$50/month in mis-configuration incidents. ROI: moderate. **Implementation:** Single `StrategyConfig(BaseModel)` with `model_config = {"frozen": True, "extra": "forbid"}`. Every field defaults to its current hardcoded value — first commit must change zero behavior. Use `Field(2.0, gt=0, le=10)` bounds so `climax_vol_mult: 200` fails at startup. Load with plain `yaml.safe_load(path.read_text())` + `StrategyConfig.model_validate(data)` — do not add `pydantic-settings` or `dynaconf` or `hydra` for a single-env bot (Hydra is designed for ML sweeps, not services; Dynaconf's multi-env is overkill). Kill the orphan `regime_matrix.yaml` and `regime_params.yaml` by folding their contents into `strategy.yaml` or deleting them outright.

**P17. Thread-safe config hot-reload (atomic reference swap).** Severity: MEDIUM. Sources: Codex v2 (dashboard race). Est: 2h. Capital impact: dashboard slider mid-evaluation = inconsistent thresholds applied to one decision. Rare but hard to debug. ~$10–$30/month. ROI: moderate. **Implementation:** Module-level `_current: StrategyConfig` (frozen Pydantic model). Readers call `cfg = config_state.get()` once at start of each bar/tick evaluation and use the snapshot throughout — CPython GIL guarantees single-name rebinding is atomic (PEP 703 preserves this even on free-threaded 3.13t). Writers (dashboard, watchdog YAML reloader) call `config_state.publish(new_cfg)` which takes a `_write_lock` only against other writers; readers never block. No `asyncio.Lock` or `threading.RLock` in the hot path → no priority inversion. A `watchdog.Observer` watches `config/strategy.yaml`; on `on_modified` reparse and `publish()`; exceptions are logged and leave old config intact. Debounce watchdog events within 500ms (editors save via tempfile-rename generating multiple events on Windows).

**P21. Pre-commit + CI hooks + pytest as canonical test runner.** Severity: MEDIUM (elevated from v1 scope). Sources: Claude Code + 2026-04-18 migration finding. Est: 2h (up from 1.5h in v1). Capital impact: indirect; prevents ~2 bugs/month from reaching prod. Additionally closes a ~10% hidden-test-discovery gap found during migration. ROI: high. **Implementation — expanded from v1 based on migration findings:** The migration Step 5 testing discovered that `python -m unittest discover tests/ -v` runs 61 tests and passes, but `python -m pytest tests/` runs 162 tests including 6 that fail (all pre-existing, traced to strategy rework drift). The 6 failing tests use pytest fixtures (`tmp_path`, `monkeypatch`, `caplog`) invisible to `unittest discover`. **Net: actual test confidence has been ~62% of reported (101/162 passing, not 61/61).** Three sub-patches: (a) **Standardize on pytest.** Change all test-runner invocations (docs, CI, pre-push hooks, CLAUDE.md instructions) from `unittest discover` to `pytest tests/ -v`. This is a one-time swap; the 61 unittest-discoverable tests also run under pytest (pytest is a superset). (b) **Fix the 6 hidden stale tests.** Task already spawned during migration — don't duplicate; track via the spawned task ID. (c) **Build the hook infrastructure:** `.pre-commit-config.yaml` with pre-commit-hooks (trailing-whitespace, end-of-file-fixer, **check-yaml**, mixed-line-ending `--fix=lf`, debug-statements), ruff-pre-commit (replaces Black+isort+flake8+pyupgrade in one Rust binary, ~100–200× faster), mypy-mirrors scoped to `src/`. Install `pre-commit install && pre-commit install --hook-type pre-push`. Put `pytest -m "not slow" -x -n auto` on **pre-push not pre-commit** — solo-dev trick to keep commits fast. GitHub Actions on `windows-latest` (matches prod OS). ## Section 4 — Explicitly OUT OF SCOPE (and why)

**A $300 sim account running a solo-dev personal system has no business adopting enterprise infrastructure.** The following are commonly-recommended by evaluations/AI tools and should be actively declined:

- **Time-series DB (InfluxDB, TimescaleDB, Prometheus + Grafana stack).** Claude Code mentioned "no metrics collection / time-series DB." Correct description, wrong remediation. For ~10 trades/day, SQLite tables + occasional pandas plots from Jupyter are sufficient for months. Installing and maintaining a TSDB is 10+ hours with ongoing config burden; the signal you'd extract is identical to what 50-line rolling-window code produces. **Revisit only if/when the bot produces >500 trades/day.**

- **Kubernetes / Docker orchestration / multi-machine deployment.** The bot runs on a Windows trading PC next to NinjaTrader. Containerizing doesn't help; NT8 is Windows-native and can't be containerized. Any "cloud-deploy your trading bot" advice from generalist LLMs is wrong for this architecture.

- **Institutional data feeds (Databento Plus/Unlimited, CQG, TT, Refinitiv Elektron, Bloomberg B-Pipe, CME DataMine subscription).** Ranges from $1,400/mo to $10k+/mo. Databento pay-as-you-go ($125 free credits, typically <$10 for a 2-month MNQ backtest corpus) covers this need entirely. Spending >5% of account equity on data is a red flag unless the strategy is demonstrably profitable.

- **Sophisticated change-point detection in production.** Research surfaced BOCPD (Adams & MacKay 2007), ruptures PELT, Bayesian online change-point. At 50-trade rolling windows these are noise-dominated. CUSUM and hard thresholds beat them every time at this sample size. Use `ruptures.Pelt` **only** for post-hoc analysis ("where did the regime break?"), never in the hot alerting path. **Multiple-testing correction matters more than algorithmic sophistication.**

- **Hydra, OmegaConf, full Dynaconf multi-env layering.** Hydra is designed for ML experiment sweeps with CLI-driven composition, not services. Dynaconf's multi-env superpower is irrelevant to a one-machine personal bot. Pydantic v2 `BaseModel` with `frozen=True` + plain `yaml.safe_load` + `model_validate` is 30 lines and has zero runtime surprises. Do not add these until you actually have a second deployment environment.

- **Rewriting onto nautilus_trader, backtrader, or Zipline.** These are excellent references for *patterns* (`Strategy`/`Actor` separation, `Cerebro.run`, `LiveClock`/`TestClock`) but the integration cost with NT8 (which speaks OIF files and ATI, not a FIX API) is weeks of work for negligible gain over a $300 sim account. Borrow the patterns, skip the framework.

- **Full Sentry paid tier, Datadog, New Relic.** Sentry's free tier (5K errors/month, 1 user, 30-day retention) is more than sufficient for a solo developer. DSN daily rate-limit (~170 events/day) protects the monthly quota against a bad deploy. Datadog and New Relic start at $31/host/mo and assume cloud infrastructure.

- **Replacing NT8 with Interactive Brokers + `ib_insync`.** Tempting because ib_insync has a cleaner API and better ACK story. But this is a rebuild, not a fix. NT8 ATI works and has a documented `outgoing\<orderId>.txt` fill-ACK channel (see P2). Keep NT8; fix the integration.

- **`atomicwrites` Python package.** Deprecated by its maintainer in 2024 with the note "Python 3 has os.replace and os.rename which probably do well enough." Pytest removed it; Spyder is tracking removal. Do not adopt. Use stdlib `tempfile` + `os.fsync` + `os.replace` directly.

- **Redis, Celery, RabbitMQ for task queues.** `asyncio.Queue` is sufficient. One extra service-dependency on a Windows trading PC is one extra thing that can hang NT8.

- **Reinforcement learning / deep learning strategy research.** The bot is losing money at 33% WR with classical setups. Adding model complexity to losing strategies makes P&L noisier, not better. **Solve the plumbing, measure the strategies cleanly, then — if edges don't emerge from the existing three strategies in clean backtests — consider a new generation of strategies.** Not models.

- **24/7 operation / auto-restart services / Windows Service deployment.** MNQ is most liquid 09:30–16:00 ET. Running the bot 24/7 increases surface area for little marginal opportunity. A Task Scheduler entry for 09:00 ET start + graceful 16:15 ET shutdown is right-sized.

- **Return to OneDrive for any NT8-adjacent files.** This is a non-negotiable addition to the out-of-scope list after the 2026-04-18 migration. If Windows Update re-enables Documents backup (which it has been known to do), **always decline**. If you want cloud backup, use a non-sync-based solution (Backblaze, Arq, scheduled robocopy to external drive) that doesn't hold file handles during copy.
