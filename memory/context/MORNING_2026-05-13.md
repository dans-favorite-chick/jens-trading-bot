## What was wrong (forensically confirmed 2026-05-12 evening)

The watchdog log captured **nine consecutive restart attempts** between 21:32 and 21:42, each spawning a dashboard-managed subprocess that died 2-3 minutes later:

```
21:32:51 Restart command sent — PID=52072   (zombie)
21:33:26 Restart command sent — PID=109540  (zombie)
21:33:58 Restart command sent — PID=63884   (zombie)
21:34:30 Restart command sent — PID=195256  (zombie)
21:35:02 Restart command sent — PID=10680   (zombie)
21:36:05 Restart command sent — PID=204576  (zombie)
21:38:08 Restart command sent — PID=103192  (zombie)
21:42:11 Restart command sent — PID=207044  (zombie)
```

Three bugs, all of them in the launch + restart code paths:

| # | Where | Bug |
|---|---|---|
| 1 | `dashboard/server.py::_start_bot` | `creationflags=CREATE_NEW_PROCESS_GROUP` caused subprocesses to die silently 2-3 min after launch (Windows + asyncio ProactorEventLoop incompatibility with detached console) |
| 2 | `dashboard/server.py::_stop_bot` | Unconditional psutil scan + `terminate()` killed ANY `python prod_bot.py` by name, including operator's `launch_prod.bat` launches |
| 3 | `tools/watcher_agent.py::_execute_restart` | Same as #1 plus `stdin=subprocess.DEVNULL` — produced even faster zombie death (~15s) |

Together they meant: any operator launch of prod_bot got killed by the next watchdog/watcher restart cycle, and the dashboard's "restart" produced a process that died within minutes, looping forever.

## What was done — commit `8b471af`

### Fix 1 — `dashboard/server.py::_start_bot`

```python
# was: creationflags=subprocess.CREATE_NEW_PROCESS_GROUP if sys.platform == "win32" else 0
# now: creationflags=0
```

Trade-off accepted: `_stop_bot`'s graceful `CTRL_BREAK_EVENT` path becomes a plain `terminate()` (hard kill on Windows). `tick_aggregator.save_state()` persists state on every bar — losing ≤ a few seconds of state on stop is acceptable.

### Fix 2 — `dashboard/server.py::_stop_bot(name, force=False)`

```python
# Path 2 (psutil scan by name) now gated on `force=True`.
# /api/bot/stop reads `force` from JSON body, defaulting False.
# Watchdog restart cycles default to force=False — never kill external launches.
# Operator UI "Stop Bot" button can opt in to force=True.
```

### Fix 3 — `tools/watcher_agent.py::_execute_restart`

Removed both `creationflags=0x00000200` (CREATE_NEW_PROCESS_GROUP) **and** `stdin=subprocess.DEVNULL`. Now uses `creationflags=0` with no explicit stdin override.

### Two observations worth investigating (not blockers)

1. **Threshold may be too low for active regimes**: filling the 50-print buffer within seconds means `threshold_contracts=25` is catching a lot. RTH may need 50 or 100 to filter retail aggregation. Tunable via `TapeReader(threshold_contracts=...)` constructor.

2. **`vol` field semantics worth checking**: 200+ contract single ticks are unusually large for MNQ. Possible NT8 is sending aggregated bar volume rather than per-trade size on some tick subtypes. Worth a forensic look at the raw tick payload to confirm the side classifications are meaningful — if `vol` is aggregated, side classification at "this aggregator's last price" may not match the actual trade-flow direction.

Both are tuning concerns for the data-collection phase, not correctness bugs.
