## What was wrong

### Silent gate #1 — `REGIME_CONFIG` allowed_strategies whitelists

In `core/session_manager.py`, the prod `REGIME_CONFIG` was silently filtering OUT most strategies in 4 of 8 regimes:

| Regime | Time window | OLD allowed_strategies | Effect |
|---|---|---|---|
| `OVERNIGHT_RANGE` | 22:00-07:00 CT | `["spring_setup"]` only | spring_setup retired 2026-04-24 → effectively ALL strategies blocked overnight |
| `PREMARKET_DRIFT` | 07:00-08:30 CT | `["bias_momentum"]` only | opening_session.premarket_breakout (08:30-08:45) and every other strategy blocked premarket |
| `AFTERNOON_CHOP` | 10:00-13:30 CT | 5-strategy whitelist | **opening_session, orb, noise_area, vwap_band_pullback, vwap_band_reversion, footprint_cvd_reversal blocked for ~3.5 hours every trading day** |
| `AFTERHOURS` | overnight | `["spring_setup"]` only | retired → all blocked |

These weren't documented as gates — they were tucked away in a regime-config dict, and the bot's eval loop quietly `SKIP_REGIME`'d non-whitelisted strategies without surfacing them anywhere obvious.

**Fix**: set `allowed_strategies = None` (all allowed) for those 4 regimes. AFTERNOON_CHOP keeps its elevated `min_confluence_override=4.0` + 0.5× size — that's the quality bar now, not an exclusion list. Strategies retain their own intrinsic time-window guards.

### Silent gate #2 — stale `prior_day_*` after multi-day uptime

`SessionLevelsAggregator.load_prior_day()` was called once at bot startup. On a fresh aggregator, no in-class daily refresh existed.

Important context discovered while debugging: there IS a separate bot-level daily refresh that fires at 00:00 CT each day (visible in the prod_bot stdout log every day from 05-07 through 05-11). That mechanism was working. So this was a defense-in-depth gap, not a live production issue — but if the bot-level refresh ever stopped firing, opening_type classification would silently break on day 2+.

**Fix**: `_maybe_daily_reset` now also calls `load_prior_day(target_date=today)` on day-rollover. Belt + suspenders.

## Open Issue #1 — dashboard `_start_bot` produces zombie subprocesses

**Symptom**: any process launched via dashboard's `/api/bot/start` (or anything that calls `subprocess.Popen` with `CREATE_NEW_PROCESS_GROUP` on Windows) starts cleanly, runs for 2-3 minutes, dies silently with no traceback. PowerShell `Start-Process -WindowStyle Hidden` has the same failure mode. The bot's `Popen` flag in `_start_bot` is:

```python
creationflags=subprocess.CREATE_NEW_PROCESS_GROUP if sys.platform == "win32" else 0
```

**Hypothesis (un-confirmed)**: ProactorEventLoop on Windows requires a properly-attached I/O environment. With `CREATE_NEW_PROCESS_GROUP` + redirected stdout + closed stdin, some async operation a few minutes in fails the event loop. The error never reaches the redirected stderr because by the time the process is dying, the file handle has been torn down.

**Repro**:
1. `curl -X POST http://127.0.0.1:5000/api/bot/start -d '{"name":"prod"}'`
2. Wait 30 seconds — bot connects to bridge, starts logging
3. Wait 2-3 more minutes — bot vanishes from process table, log goes silent, no traceback

**Workaround used tonight**: launch via Bash background or foreground (inherits Bash's console/stdin). Bash bg currently keeps the bot alive: task `brziw463g` in my Claude session.

**Action for today**:
1. Investigate by removing `CREATE_NEW_PROCESS_GROUP` from `dashboard/server.py::_start_bot`. The flag was added so `send_signal(CTRL_BREAK_EVENT)` works for graceful shutdown. Replacing it with `creationflags=0` + a different graceful-shutdown mechanism (e.g., an HTTP /shutdown endpoint inside the bot) would be the structural fix.
2. Until fixed, **manual relaunch via `launch_prod.bat`** is the only reliable restart. Watchdog auto-restart is compromised because it routes through the broken dashboard path.
